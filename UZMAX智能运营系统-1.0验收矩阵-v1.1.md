# UZMAX 智能运营系统 1.0 验收矩阵 v1.1

> 状态：1.0 发布阻断清单修订版  
> 日期：2026-06-13  
> 规则：任何 P0 阻断项未通过，1.0 不得正式发布。M0-M6 是内部推进闸门，GA-0 是生产内测数据闸门，都不是外部分期发布。

---

## 1. 验收分级

| 等级 | 含义 | 发布影响 |
|---|---|---|
| P0 | 安全、隔离、生产闭环、核心运营能力 | 任一失败阻断 1.0 |
| P1 | 重要体验、效率、可观测完整度 | 可限期修复，但需 owner 签字 |
| P2 | 增强体验、低频便利能力 | 可进入 1.0 后修复池 |

证据产生者用于 owner 排期：`自动化` 指 CI/E2E/评测可自动产生证据；`实现方` 指开发/运维提供演练或配置证据；`owner` 指需要业务负责人输入、盲评或签收；`外部 spike` 指必须在真实第三方环境验证。

## 13. 条件式 Spike 通则

1. SPK-01 Telegram Business 截止 M2 闸门内，结论写 ADR-B01。全部可行时 C-03/C-04/C-05 为 P0；部分可行时 C-05 降为 P1 并启用 C-05b；不可行时 C-03-C-05 移出阻断清单，C-03b 成为 P0。
2. SPK-02 订单 SaaS API 截止 M4 前，结论写 ADR-B02。有 API 时 E-01 为 P0；无 API 或不可用时 E-01 移出阻断清单，E-02 导入兜底升级为 P0 主路径，E-03/E-04 仍为 P0。
3. SPK-03 RLS x Prisma x 连接池、SPK-04 双鉴权链路必须在 M0 完成，结论分别写 ADR-001、ADR-002。
4. spike 必须在真实环境完成，不接受纯 mock 结论。
5. 截止闸门到期仍无结论时，自动按不可行分支执行；默认保守，不允许无限挂起发布定义。
6. 分支切换必须通过配置与 feature flag 完成，不得在发布末期靠代码重构临时改口径。

## 14. 发布判定

1. 所有当前分支下的 P0 项必须通过，并保存证据链接。
2. P1 未通过项必须有 owner 明确签字、修复截止日期和风险说明。
3. P2 未通过项进入 1.0 后修复池，不阻断发布。
4. 每个里程碑关闭时必须归档该阶段验收证据；M6 只做全链路演练与残项清零。
5. 正式发布前必须完成一次全链路故障演练：模型全挂、订单 API 挂或导入主路径异常、worker 积压、AI 熔断、RLS 越权尝试。
