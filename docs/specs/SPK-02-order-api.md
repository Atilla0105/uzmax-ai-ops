# SPK-02 订单 SaaS API 真实环境 Spike

## 目标

验证订单 SaaS 是否提供可支撑 1.0 只读订单查询的 API，并将结论写入 `docs/adr/ADR-B02-order-api.md`。本 spec 只做 spike 与分支判定，不实现完整订单中心。

## Owner

Owner：AI agent 执行 API spike；项目 owner 提供 API 文档/沙箱凭据或确认无 API，并确认分支结论。

## 时间盒

拿到 API 文档/沙箱凭据或书面无 API 确认后 1 个工作日；字段、限流或新鲜度争议最多顺延到 2 天并写阻断记录；硬截止不得晚于 M4。

## 触碰模块/文件

- 临时 API 调用脚本或 Postman/HTTP collection。
- 字段映射表。
- `docs/adr/ADR-B02-order-api.md`。
- `docs/evidence/M4/spikes/SPK-02-order-saas-api/`。

## 前置条件

- 真实订单 SaaS API 文档或沙箱凭据；若无 API，需项目 owner 明确确认。
- 样例订单号、批次号、客户标识。
- 可对比订单 SaaS 前台或项目 owner 提供的真实状态。
- CSV/表格导入样例。

## 实施步骤

1. 验证鉴权方式、token 生命周期、IP 白名单、租户/店铺范围。
2. 按订单号、批次号、客户标识分别查询。
3. 记录字段覆盖：外部 ID、状态、批次、更新时间、过期时间、物流/仓储状态、原始 payload 摘要。
4. 验证增量同步、分页、频率限制、超时、429/5xx、鉴权失败。
5. 抽样对比真实状态，确认数据新鲜度和字段含义。
6. 验证 API 不可用时导入快照能作为主路径或降级路径。
7. 汇总请求/响应摘要、字段映射、限流记录、错误样本、新鲜度对比、导入样例。

## 通过条件

- API 存在且沙箱可用。
- 鉴权稳定。
- 查询维度覆盖订单号、批次号或客户标识中的核心客服场景。
- 字段和新鲜度能支撑对客只读查询。
- ADR-B02 对 E-01 / E-02 / E-03 / E-04 给出明确分支。
- ADR-B02 记录 API 文档版本、凭据范围、测试数据来源、鉴权方式、字段映射、频率限制、数据新鲜度、错误模型和主路径/辅助路径/关闭判定。

## 失败分支

- 有 API 且可用：E-01 为 P0，connector 成为订单查询主路径。
- 部分可用：API 仅作辅助或灰度能力，导入快照成为实际主路径或阶段主路径。
- 无 API、不可用或到期无结论：E-01 移出阻断清单，E-02 导入快照升级为 P0 主路径，E-03/E-04 仍为 P0。

## 不做什么

- 不修改订单、支付、财务、仓库状态。
- 不实现完整订单页面。
- 不把不可靠 API 包装成 P0 已通过。
- 不让 AI 编造物流或订单状态。

## 验收映射

- E-01 / E-02 / E-03 / E-04。
- 条件式 Spike 通则 §13。
